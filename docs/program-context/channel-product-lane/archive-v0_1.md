## 3. Registry consolidation

This archive preserves the lane registry sections 2, 3, 3.11, 4, 5, 6, 7, and 8 in consolidated form.

### 3.1 Section 2 — Program-context placement

[D] The channel-product lane is a program-context lane, not a theorem-track lane.

[P] Its purpose is to capture how channel products, shared-edge obstructions, and multiplicity frontiers inform the strong-coupling program without modifying Lane I.

[L] This lane does not change the v0.14.4 fixed-spacing SU(2) theorem statement; does not alter `M0`, `Dinc`, `a*`, `qKP`, `beta*`, or the KP certificate; and does not claim a continuum mass gap.

[I] The lane isolates a recurring obstruction pattern: scalar channel-product closure can hold in restricted/trivalent settings while higher-valent or cyclic settings force matrix-valued, basis-sensitive, or operator-valued replacements.

### 3.2 Section 3 — Program A: channel-product core

[D] Program A records the channel-product theorem pattern in the scalar/trivalent regime.

[T] Channel-product theorem pattern, scoped: under scalar/trivalent hypotheses, channel data may compose through a product-like aggregation rule after normalization choices are fixed.

[A] The scalar/trivalent hypotheses are load-bearing. The theorem pattern assumes no nontrivial intertwiner multiplicity or else a basis/normalization has already suppressed it.

[C] Program A has enough structure to close this codification tranche: it has a channel-product pattern, a known product-failure boundary, and an identified higher-valent frontier.

[L] Program A does not establish arbitrary cyclic Frobenius closure, arbitrary spin-network channel aggregation, or transfer-operator spectral-radius control. Those are Lane VII or future proof-target obligations.

### 3.3 Section 3.11 — Phase 0.5 frontier

[D] Phase 0.5 is the first frontier after the Program A/B scalar codification freeze.

[P] Phase 0.5 is a design layer, not a theorem. It is required before higher-valent channel-product or product-failure claims can be stated cleanly.

Phase 0.5 must define:

1. [D] Intertwiner basis.
2. [D] Bracketing convention / coupling tree.
3. [D] Multiplicity index `alpha`.
4. [D] Recoupling matrices between admissible bases.
5. [A] Intertwiner normalization.
6. [P] Matrix-valued product-failure test replacing the scalar determinant when multiplicity is nontrivial.

[L] Phase 0.5 is not optional once the lane moves beyond trivalent/scalar examples. A scalar determinant in a multiplicity-bearing setting is under-specified.

### 3.4 Section 4 — Program B: product-failure/shared-edge lane

[D] Program B records product-failure behavior in shared-edge / shared-screen configurations.

[T] Product-failure determinant pattern, scoped: in scalar or trivalent settings, product separability can fail in a way detected by a determinant-like obstruction.

[A] The scalar determinant formulation assumes no hidden multiplicity index. In higher-valent settings, product failure must become basis-aware and likely matrix-valued.

[C] The shared-edge trivalent hinge amplitude and scalar character product-failure lemma remain valid only at their scoped level.

[E] Charge-mode U(1) control remains a useful abelian calibration. It does not prove nonabelian transfer closure.

[L] Program B does not yet contain a transfer operator and does not yet contain arbitrary higher-valent multiplicity handling.
